# Project-Based Learning Bot — Fidelity Checklist

**Evaluation unit note**: unlike the other checklists in this family, most items here cannot be scored from a single transcript. Score across the full set of session transcripts for one project (launch session + all return check-ins + closing session). A single session can only fairly be scored on the items marked **[single-session]**; everything else requires the full arc.

Each item scores **Pass / Partial / Fail**, with a note citing the specific session and line(s).

---

## Launch Session

**L1. Question before artifact [single-session]**
Did the session establish a driving question before locking in what the artifact would be, rather than jumping straight to "what will you build"?
- Pass: question discussion precedes or co-develops with artifact discussion
- Fail: artifact gets decided with no real driving question established

**L2. Question is real, not a topic [single-session]**
Is the resulting driving question actually a question (open-ended, answerable through investigation) rather than a topic or assignment restated as a sentence?
- Pass: question is genuinely open-ended and couldn't be answered with a single fact
- Fail: "question" is a topic with a question mark added ("What is climate change?" rather than "What should our town do about flooding?")

**L3. User helped shape the question, not just received it [single-session]**
Did the user have visible input into the driving question, rather than the bot simply assigning one?
- Pass: question shows evidence of co-development or user origination
- Fail: bot hands the user a fully-formed question with no real shaping input from them

**L4. Authenticity check actually happened [single-session]**
Did the bot explicitly check whether the project connects to something real (real audience, real use, real problem) rather than skipping straight past this?
- Pass: explicit authenticity discussion occurred
- Fail: no authenticity check occurred at all

**L5. Artifact is concrete [single-session]**
Is the artifact specific enough to know when it's done (a defined thing, not "learning about X")?
- Pass: artifact has a recognizable, checkable form
- Fail: artifact remains vague enough that "done" wouldn't be identifiable later

## Return Check-Ins (score across all return sessions in the project)

**R1. Check-in happens before coaching**
At the start of each return session, does the bot ask the user to restate what happened since last time before responding to content?
- Pass: check-in precedes content response in every return session
- Fail: bot responds to content immediately without a restate step in one or more sessions

**R2. No false continuity**
Does the bot avoid asserting specific claims about "what happened last time" that weren't actually restated by the user in this session? (Cross-check against the prior session's actual transcript if available.)
- Pass: bot's claims about prior progress are sourced from the user's current restatement, not invented
- Fail: bot states something about prior progress that doesn't match what the user said (in this or the prior transcript)

**R3. Stalled-project signal handled honestly**
If the user struggles to articulate where they left off, does the bot name that openly rather than plowing ahead as if nothing's off?
- Pass: bot notices and names vague/absent recall
- Fail: bot proceeds as though a clear handoff occurred when it didn't
- N/A: if recall was clear in every session, this item doesn't apply

**R4. Coaching is specific to the actual work described**
Does the bot's response engage with the specific research/draft/attempt the user describes, rather than generic encouragement?
- Pass: response cites specifics from what the user just described
- Fail: response is generic enough to apply to any project update

**R5. Bot does not do the work**
Across all return sessions, does the bot avoid producing the actual research, draft, or build the user should be producing themselves — even when directly asked for a shortcut?
- Pass: bot redirects requests to do the work back to the user, offering guidance rather than substitution
- Fail: bot produces substantial actual project content (a draft section, the research itself) in the user's place

---

## Iterative Inquiry (score across the project as a whole)

**I1. The driving question is allowed to evolve**
Across sessions, if the user's understanding shifted, did the bot treat a revised question as legitimate rather than redirecting back to the original?
- Pass: bot accommodates a genuine shift in the question when one occurs
- Fail: bot insists on the original question even after the user's findings suggest a better one
- N/A: if the question never shifted and didn't need to, this item doesn't apply

**I2. Next steps originate from the user's curiosity, not bot assignment**
Across the "refine next iteration" portions of return sessions, do next steps emerge from what the user expressed curiosity about, rather than being assigned by the bot?
- Pass: next-step language is clearly drawn from the user's own expressed interest or confusion
- Fail: bot regularly hands down next steps as assignments

---

## Closing Session

**CL1. Real close, not another check-in**
Does the closing session function differently from a return check-in — treating the artifact as genuinely finished rather than another "what's next" loop?
- Pass: closing session is recognizably distinct in tone and content from return sessions
- Fail: closing session reads like one more check-in with no real sense of ending

**CL2. Artifact explicitly connected back to the driving question**
Does the user (prompted by the bot) explicitly state how the finished artifact answers or responds to the original driving question?
- Pass: explicit connection is made
- Fail: artifact and driving question are never explicitly reconnected at the close

**CL3. Retrospective reflection occurred**
Did the bot ask what the user would do differently knowing what they now know — the accumulated-learning question?
- Pass: this question or its equivalent was asked and answered
- Fail: closing session ends without this reflection

**Hard failures worth flagging in isolation:**
- R2 Fail (false continuity) — undermines the honesty the whole multi-session design depends on; the bot claiming to remember things it's actually inventing is a worse failure than admitting it doesn't know
- R5 Fail (bot does the work) — the most direct violation of the family's core thesis, and easiest to slip into across a long project when the user is tired or behind
- L4 Fail (no authenticity check) — per the literature, this is arguably the single most load-bearing element of real PBL; skipping it at launch is hard to recover from later

## Notes for Use
- This checklist is unusable against a single session — don't attempt to score L/R/I/CL items from one transcript alone; collect the full project arc first.
- R2 specifically requires comparing what the bot says against what actually happened in a prior transcript, not just internal consistency within the current session — this is the one item in the whole family of checklists that needs cross-session fact-checking rather than within-session judgment.
- Expect I1 and R3 to be N/A in many real runs; only mark them Fail if the triggering condition (a shift, or a vague recall) actually occurred and was handled badly.
